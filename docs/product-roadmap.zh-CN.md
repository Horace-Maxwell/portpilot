# PortPilot 产品路线图

PortPilot 的目标应该是把“我刚发现一个仓库”尽快变成“它已经在本地跑起来，而且我能轻松管理它”。

这份路线图参考了本地开发环境工具、服务面板、反向代理、容器管理界面和开发环境产品里的成熟思路，例如 Portainer、Homepage、DevPod、GitHub Codespaces、DDEV、ServBay 和 Local。

## P0 — 把第一次跑起来做得更傻瓜

- 增加 `Doctor` 检查，在首次运行前发现缺失运行时、版本不匹配、端口占用和系统工具缺失。
- 把仓库导入做成完整向导：clone、安装依赖、填写 env、运行、分配路由、打开页面。
- 强化 monorepo 检测，让 PortPilot 能识别真正该启动的子应用，而不是只猜一个根命令。
- 推断时读取更多信号：README 片段、devcontainer 文件、`.nvmrc`、`.node-version`、`.python-version`、Docker 文件和常见框架配置。
- 为常见失败提供一键修复动作，比如缺依赖、env 缺值、端口冲突。

## P1 — 把日常重复操作全部省掉

- 保存并恢复 workspace session，一键恢复整组项目的运行态。
- 支持批量动作：全部启动、全部停止、重启异常项目、恢复上次会话。
- 把运行时状态细分得更清楚：安装中、启动中、等待健康检查、运行中、降级、失败。
- 升级日志体验：搜索、错误过滤、告警置顶、按运行实例查看历史。
- 把本地进程和 Compose 服务统一进一个 Runtime 面板，让它们看起来是同一个工作台的一部分。

## P1 — 让路由与预览更有用

- 从日志和监听端口中自动识别真实访问地址，而不是主要依赖手工提示。
- 强化 route 健康检查和 ready 轮询。
- 为每个项目突出一个主入口地址，只在必要时展示备选地址。
- 增加一键分享预览，把本地 route 临时暴露为可对外访问的链接。

## P2 — 让项目更容易团队复用

- 引入项目级配置文件，比如 `.portpilot.json`，固定推荐动作、端口、env 提示和健康检查。
- 支持导出 / 导入可复用的项目 recipe。
- 支持更好的 env 模板共享能力，包括必填字段和脱敏说明。
- 当仓库已经有 devcontainer 或 DevPod 元数据时，优先复用这些启动信息。

## P2 — 让平台更容易扩展

- 为更多语言和框架增加可插拔的动作推断器。
- 提供自定义健康检查和日志解析扩展点。
- 加入常见技术栈模板包，例如 Next.js、Vite、Django、FastAPI、Tauri 和全栈 Docker 项目。
- 在不破坏本地优先定位的前提下，为远程运行时和云工作区预留可选集成。

## 建议补充的类型

- `DoctorCheck`
- `WorkspaceSession`
- `ProjectRecipe`
- `DetectedAppTarget`
- `TunnelSession`

## 产品标准

- 每个新能力都应该至少减少一个重复的人肉步骤。
- 核心流程必须兼容 macOS、Windows 和 Linux。
- “导入后成功跑起来”和“运行时一眼看明白”比功能数量更重要。
